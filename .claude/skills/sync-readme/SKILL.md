# /sync-readme — синхронизация README перед коммитом

Скилл для проекта **tg_agg**. Цель — чтобы [README.md](../../../README.md) не
отставал от кода: перед коммитом находим, что изменилось, и приводим README в
соответствие. Меняем README только по делу — если документируемое поведение не
затронуто, ничего не правим.

## Шаги

### 1. Понять, что изменилось

Собери полную картину правок (и staged, и нет):

```bash
git diff --staged --stat
git diff --stat
git status --short
```

Затем посмотри содержательный диф по файлам, которые влияют на документацию
(см. карту ниже), через `git diff [--staged] -- <путь>`.

### 2. Определить затронутые разделы README

Сопоставь изменения с разделами README. Карта «что в коде → что в README»:

| Изменилось в коде | Раздел README для проверки |
|---|---|
| Команды бота — [app/bot/handlers.py](../../../app/bot/handlers.py) (`@router.message(Command(...))`, тексты `HELP`) | таблица **«Команды бота»** |
| Переменные окружения — [app/config.py](../../../app/config.py), [.env.example](../../../.env.example) | блоки про `.env`, **«Подготовка»**, разделы запуска |
| Деплой/инфра — [deploy/](../../../deploy/), [docker-compose.yml](../../../docker-compose.yml), [Dockerfile](../../../Dockerfile) | **«Запуск через Docker»**, **«Авто-деплой на сервере (cron)»** |
| Новые/удалённые модули в [app/](../../../app/) | дерево в разделе **«Структура»** |
| Поток обработки — [app/pipeline.py](../../../app/pipeline.py) | нумерованный список шагов в начале README |
| Стек/зависимости — `pyproject.toml` | раздел **«Стек»** |

Если ни один пункт не задет (например, правка только в тестах или внутренний
рефакторинг без изменения поведения) — скажи об этом и переходи к шагу 5,
README не трогаем.

### 3. Обновить README

Правь точечно, сохраняя стиль документа:

- Язык README — **русский**, как и весь файл; код/команды — как есть.
- Не дублируй и не переписывай разделы целиком — меняй только нужные строки.
- Команды бота, переменные ENV и шаги деплоя должны **дословно** совпадать с
  кодом (имена команд, имена переменных, дефолты).
- Ссылки на файлы — в markdown-формате `[файл](путь)`, как в остальном README.

### 4. Показать результат

```bash
git diff -- README.md
```

Кратко перечисли, что обновил и почему (какое изменение в коде это отражает).
Если README не требовалось трогать — так и скажи.

### 5. Передача дальше

- Этот скилл **только** синхронизирует README — он не запускает тесты и не
  коммитит сам.
- Дальше логично прогнать `/check` (тесты/линтеры) и через него же
  закоммитить — README уже будет в стейдже вместе с правками кода.
- README добавляй в тот же коммит, что и связанные изменения кода, чтобы
  история оставалась консистентной.

## Подводные камни проекта

- README на русском — не переключай язык и не «улучшай» формулировки соседних
  разделов без необходимости.
- `.env` в `.gitignore` — документируем только `.env.example`, не реальные
  значения.
- Скилл реактивный: он не срабатывает сам по таймеру. «Перед коммитом» означает
  «вызови его перед тем, как коммитить». Если нужна жёсткая автоматизация на
  каждый commit — это уже git-хук (`.git/hooks/pre-commit`) или хук харнесса в
  `settings.json`, а не скилл.
