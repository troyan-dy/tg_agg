# /publish — публикация изменений с учётом риска

Скилл для проекта **tg_agg**. Цель — закрыть «последнюю милю»: довести готовую
правку до `git push`, но по-разному в зависимости от того, что менялось.

Два режима, выбираются автоматически по содержимому диффа:

- **Код затронут** → нужно убедиться, что не сломали: гоняем проверки и
  **предлагаем** опубликовать (ждём явного «да»).
- **Код не затронут** (только доки/конфиг/скиллы) → ломать нечего: коммитим и
  пушим **сами**, без вопроса.

## Шаг 1. Собрать и классифицировать изменения

```bash
git status --short
git diff --stat HEAD
```

Если рабочее дерево чистое — публиковать нечего, так и скажи, на этом всё.

Иначе получи список затронутых файлов и определи режим:

```bash
git status --short | awk '{print $2}'
```

**«Код затронут»**, если хоть один файл попадает в рантайм/проверяемое:

- `app/**`, `tests/**`
- `migrations/**`, `alembic.ini`
- `pyproject.toml`, `uv.lock` (зависимости влияют на прод)
- `Dockerfile`, `docker-compose.yml`

**«Код не затронут»** — когда ВСЕ файлы только из этого списка:

- `README.md` и прочие `*.md` (включая `CLAUDE.md`)
- `.claude/**` (скиллы, агенты, настройки харнесса)
- `.env.example`, `.gitignore`

Сомневаешься, куда отнести файл, — считай его кодом (безопаснее спросить).
Никогда не добавляй в коммит `.env` (он в `.gitignore`): после `git add`
проверяй `git diff --cached --name-only`.

## Шаг 2А. Код затронут → проверки и предложение

1. Если правка влияла на документируемое поведение (команды бота, ENV, деплой,
   пайплайн) — сначала прогони `/sync-readme`, чтобы README не отстал.
2. Прогони `/check` (ruff + mypy + pytest с покрытием).
3. **Если хоть что-то красное** — не публикуй. Почини, перепрогони. Не глуши
   проверки и не снижай порог покрытия ради зелёного.
4. Когда всё зелёное — покажи `git diff --stat`, предложи короткое сообщение
   коммита (на русском, в стиле истории репозитория) и **спроси подтверждение**.
   Коммить и пушь только после явного «да»:

```bash
git add -A
git commit -m "<сообщение>"   # последней строкой — Co-Authored-By, см. ниже
git push
```

## Шаг 2Б. Код не затронут → публикую сам

Проверки не нужны (тесты/линтеры/типы покрывают только код). Коммить и пушь
**без вопроса**, сразу:

```bash
git add -A
git diff --cached --name-only          # убедиться, что не утянули .env/код
git commit -m "<сообщение>"            # последней строкой — Co-Authored-By
git push
```

Сообщение коммита — по сути правки, на русском, в стиле истории репозитория
(например «Обновить README под новые команды», «Добавить скилл publish»).
После пуша кратко отчитайся: что закоммитил и что запушил.

## Сообщение коммита

Всегда заканчивай тело коммита строкой:

```
Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>
```

## Подводные камни проекта

- Пакетный менеджер только **uv** — проверки гоняй через `/check`, не зови
  `pytest`/`ruff`/`mypy` напрямую.
- Дефолтная ветка для пушей — `main`. Если ты на другой ветке (например
  фиче-ветке) — пушь её же (`git push -u origin <ветка>` при первом пуше), не
  переключайся на `main` молча.
- `.env` в `.gitignore`, `deploy/*` тоже игнорируется — следи, чтобы они не
  попали в `git add`.
- Скилл **реактивный**: он запускается, когда его вызвали (вручную или как
  финальный шаг после правки). Если нужна жёсткая автоматизация «пушить на
  каждое изменение само» — это уже хук харнесса в `.claude/settings.json` (Stop
  hook) или git-хук, а не скилл. Скилл лишь кодирует решение «спросить vs
  запушить сам».
