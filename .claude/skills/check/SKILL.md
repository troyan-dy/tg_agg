# /check — тесты, линтеры и предложение опубликовать

Скилл для проекта **tg_agg** (uv + pytest + ruff + mypy). Запускается вручную
после изменений в коде. Цель — быстро убедиться, что всё зелёное, и предложить
закоммитить/запушить.

## Шаги

### 1. Все технические проверки одной командой

Из корня проекта:

```bash
make check
```

Это последовательно гоняет линтер, типы и тесты (цели `lint` → `types` → `test`
в [Makefile](../../../Makefile)):

- **ruff** — `uv run ruff check .`
- **mypy** — `uv run mypy`
- **pytest** — `uv run pytest -q` (покрытие включено в `addopts` pyproject: прогон
  сам считает `--cov=app`, печатает непокрытые строки и **падает, если покрытие
  ниже порога** `--cov-fail-under`, сейчас 90%).

`make` останавливается на первой упавшей цели. Если нужно прогнать что-то одно
после починки — есть отдельные цели `make lint` / `make types` / `make test`.

### 2. Чинить найденное

- **ruff:** попробуй автофикс безопасных правил и покажи, что починилось —
  `uv run ruff check . --fix`. Оставшиеся (не-автофиксимые, например длинные
  строки `E501`) правь вручную, сохраняя стиль окружающего кода. Не глуши
  правила через `# noqa` без причины.
- **mypy:** чинить реальные проблемы типов. Заглушки `# type: ignore[code]` —
  только с узким кодом ошибки и коротким комментарием почему (как сделано в
  `app/config.py` для pydantic-settings).
- **pytest:** разберись и почини код или тест. Не удаляй и не «скипай» тесты,
  чтобы получить зелёный прогон. Если менялось поведение — обнови/допиши тесты в
  `tests/`. Если упало именно покрытие — допиши тесты на новый код, **не** снижай
  порог, чтобы «позеленело» (порог двигаем только осознанно и отдельно).

После любой починки перепрогони `make check`, пока всё не станет зелёным.

### 3. Итог

Кратко отчитайся таблицей вида:

| Проверка | Результат |
|----------|-----------|
| ruff     | ✅ / ❌    |
| mypy     | ✅ / ❌    |
| pytest   | ✅ (N passed, покрытие M%) / ❌ |

Если хоть что-то красное — **не предлагай коммит**, сначала почини и перепрогони.

### 4. Синхронизировать README (sync-readme)

Только когда все три проверки зелёные, перед предложением коммита прогони
скилл `/sync-readme` (через Skill) — он сверит [README.md](../../../README.md) с
дифом и точечно обновит затронутые разделы (команды, ENV, деплой, структура,
поток пайплайна). Так README попадёт в тот же коммит, что и правки кода.

Если sync-readme решит, что README трогать не нужно (правки только в тестах или
внутренний рефакторинг) — это нормально, идём дальше.

### 5. Предложение опубликовать в git

Только когда все три проверки зелёные и README синхронизирован:

```bash
git status --short
```

- Если рабочее дерево чистое — скажи об этом, публиковать нечего.
- Если есть изменения — покажи `git diff --stat`, предложи короткое сообщение
  коммита (по сути правок, на русском, в стиле истории репозитория) и **спроси
  подтверждение** перед коммитом и пушем. Не коммить и не пушь без явного «да».

При подтверждении:

```bash
git add -A
git commit -m "<сообщение>"   # завершай строкой Co-Authored-By (см. ниже)
git push
```

Сообщение коммита заканчивай:

```
Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>
```

## Подводные камни проекта

- Пакетный менеджер только **uv** — не зови `pytest`/`ruff`/`mypy` напрямую.
- `.env` в `.gitignore` — следи, чтобы он не попал в `git add` (проверяй вывод
  `git diff --cached --name-only`).
- Тесты офлайн: БД — in-memory SQLite, внешние вызовы (DeepSeek, Telegram,
  feedparser) замоканы. Сеть/Postgres для прогона не нужны.
- Дефолтная ветка для пушей — `main`.
